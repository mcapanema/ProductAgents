# CI Phase Split + Security Scanning Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Split the single `test` CI job into three independent, parallel jobs — `test` (lint + format + pytest), `typecheck` (ty), and a new `security` job (pip-audit + bandit + gitleaks).

**Architecture:** The existing workflow (`.github/workflows/ci.yml`) runs every check as sequential steps inside one `test` job. We decompose it into three jobs with no `needs:` between them, so GitHub Actions runs them concurrently and each fails/passes on its own. The `security` job introduces dependency-vulnerability scanning (pip-audit against the locked production deps), static security analysis of our own source (bandit), and secret scanning (gitleaks). bandit is vendored as a dev dependency (devs run it locally; config lives in `pyproject.toml`); pip-audit runs ephemerally via `uvx`; gitleaks runs from its pinned official Docker image.

**Tech Stack:** GitHub Actions, uv (`astral-sh/setup-uv`), ruff, ty, pytest, bandit, pip-audit, gitleaks.

## Global Constraints

- Package manager is **uv** (not pip/conda). Requires **Python ≥ 3.14**. Copied verbatim from `pyproject.toml`: `requires-python = ">=3.14"`.
- Every CI job follows the existing pattern: `actions/checkout@v5` → `astral-sh/setup-uv@v8.2.0` (with `enable-cache: true`) → `uv sync --frozen` → `uv run --frozen <tool>`. Match this exactly.
- All third-party actions and the gitleaks Docker image are **version-pinned** (no `@latest`, no `:latest`). Current pins: `actions/checkout@v5`, `astral-sh/setup-uv@v8.2.0`, `ghcr.io/gitleaks/gitleaks:v8.30.1`.
- The three jobs are **parallel**: none declares `needs:`.
- Security checks are **blocking** by default (a non-zero exit from any scan fails the job). Do not add `continue-on-error`.
- There is no unit-test harness for a YAML workflow. The honest verification analog used throughout this plan is: (a) parse the YAML to prove it is valid, and (b) run each job's underlying commands locally and confirm the real exit codes/output. Steps below do exactly that.

## Task 1: Split `test` into parallel `test` + `typecheck` jobs

Pull the type-check step out of the `test` job into its own `typecheck` job. After this task the workflow has two jobs running in parallel; `test` keeps lint + format-check + pytest, `typecheck` runs `ty`.

**Files:**
- Modify: `.github/workflows/ci.yml` (currently lines 8–34, the single `jobs.test` block)

**Interfaces:**
- Consumes: nothing from other tasks.
- Produces: a `jobs.test` job (steps: checkout, install uv, sync, Lint, Format check, Test) and a `jobs.typecheck` job (steps: checkout, install uv, sync, Type check). Task 3 appends a sibling `jobs.security` job using the same checkout/uv/sync preamble these jobs establish.

- [ ] **Step 1: Replace the single-job `jobs:` block with two parallel jobs**

Overwrite the `jobs:` section of `.github/workflows/ci.yml` so the whole file reads exactly:

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Check out the repository
        uses: actions/checkout@v5

      - name: Install uv
        uses: astral-sh/setup-uv@v8.2.0
        with:
          enable-cache: true

      - name: Install dependencies
        run: uv sync --frozen

      - name: Lint
        run: uv run --frozen ruff check .

      - name: Format check
        run: uv run --frozen ruff format --check .

      - name: Test
        run: uv run --frozen pytest

  typecheck:
    runs-on: ubuntu-latest
    steps:
      - name: Check out the repository
        uses: actions/checkout@v5

      - name: Install uv
        uses: astral-sh/setup-uv@v8.2.0
        with:
          enable-cache: true

      - name: Install dependencies
        run: uv sync --frozen

      - name: Type check
        run: uv run --frozen ty check src
```

- [ ] **Step 2: Prove the YAML is valid**

Run: `uv run python -c "import yaml; d = yaml.safe_load(open('.github/workflows/ci.yml')); assert set(d['jobs']) == {'test', 'typecheck'}, d['jobs']; print('YAML OK:', list(d['jobs']))"`

Expected: `YAML OK: ['test', 'typecheck']` and exit code 0. (pyyaml is present in the project environment as a transitive dependency, so `uv run python` can import it.)

- [ ] **Step 3: Run the `test` job's commands locally**

Run: `uv run --frozen ruff check . && uv run --frozen ruff format --check . && uv run --frozen pytest`

Expected: ruff reports `All checks passed!`, format check reports the files are already formatted, and pytest finishes with all tests passing and coverage ≥ 90% (the `--cov-fail-under=90` gate in `pyproject.toml`). Exit code 0.

- [ ] **Step 4: Run the `typecheck` job's command locally**

Run: `uv run --frozen ty check src`

Expected: ty reports no diagnostics and exits 0.

- [ ] **Step 5: Commit**

```bash
git add .github/workflows/ci.yml
git commit -m "ci: split typecheck into its own parallel job"
```

## Task 2: Add bandit tooling to the project

Vendor bandit as a dev dependency with its TOML config support, and add a `[tool.bandit]` section so both CI and developers run it the same way. This task is a prerequisite for the bandit step in Task 3. pip-audit and gitleaks need no project changes (they run via `uvx` and Docker respectively in Task 3).

**Files:**
- Modify: `pyproject.toml` (the `[dependency-groups].dev` list and a new `[tool.bandit]` section)
- Modify: `uv.lock` (regenerated by `uv add` — do not hand-edit)

**Interfaces:**
- Consumes: nothing from other tasks.
- Produces: a `bandit` executable resolvable via `uv run --frozen bandit`, configured by `[tool.bandit]` in `pyproject.toml`. Task 3's "Static security analysis" step calls `uv run --frozen bandit -c pyproject.toml -r src`.

- [ ] **Step 1: Add bandit (with the `toml` extra) as a dev dependency**

Run: `uv add --dev 'bandit[toml]>=1.8'`

This appends `"bandit[toml]>=1.8"` to `[dependency-groups].dev` in `pyproject.toml`, resolves it, updates `uv.lock`, and installs it into the project environment. The `[toml]` extra is required for bandit to read its configuration from `pyproject.toml`.

Expected: command succeeds; `pyproject.toml`'s `dev` group now contains `bandit[toml]`.

- [ ] **Step 2: Add the bandit configuration section to `pyproject.toml`**

Insert this section into `pyproject.toml` (place it immediately after the existing `[tool.coverage.report]` section, before `[tool.ruff]`, to keep tool sections grouped):

```toml
[tool.bandit]
# Scan only application source in CI (`-r src`); tests legitimately use
# patterns bandit flags (e.g. asserts), so exclude them from any wider scan.
exclude_dirs = ["tests"]
```

- [ ] **Step 3: Run bandit exactly as CI will**

Run: `uv run --frozen bandit -c pyproject.toml -r src`

Expected: bandit completes with **no issues identified** and exit code 0. (Verified during planning: `src/` contains no `subprocess`, `eval`, `exec`, `pickle`, `yaml.load`, or other bandit-flagged patterns.)

- [ ] **Step 4: Run pip-audit exactly as CI will (no project change needed — this verifies the command)**

Run:
```bash
uv export --frozen --no-dev --no-emit-project --no-hashes -o /tmp/requirements-prod.txt
uvx pip-audit --no-deps -r /tmp/requirements-prod.txt
```

Expected: `No known vulnerabilities found` and exit code 0. (`--no-dev` audits only the shipped production dependency tree; `--no-deps` is correct because `uv export` already produced a fully resolved, pinned list. A harmless warning about hashes may print — it does not affect the result.)

> If pip-audit instead reports a vulnerability here, that is a real finding, not a plan error. Triage it: upgrade the offending package via `uv lock --upgrade-package <name>` if a fix version exists, or, only with explicit team sign-off, suppress a specific advisory with `uvx pip-audit --no-deps --ignore-vuln <ID> -r ...` and record why. Do not proceed to Task 3 by weakening the scan.

- [ ] **Step 5: Confirm the lockfile is consistent**

Run: `uv lock --check`

Expected: reports the lockfile is up to date (exit 0). This guarantees the Task 3 jobs' `uv sync --frozen` will succeed in CI.

- [ ] **Step 6: Commit**

```bash
git add pyproject.toml uv.lock
git commit -m "build: add bandit dev dependency and config for security scanning"
```

## Task 3: Add the parallel `security` job

Append a third job that runs the three security scans. It reuses the same checkout/uv/sync preamble as the other jobs (sync is needed so `uv run --frozen bandit` resolves the dev dependency added in Task 2).

**Files:**
- Modify: `.github/workflows/ci.yml` (append a `security:` job under `jobs:`)

**Interfaces:**
- Consumes: the `bandit[toml]` dev dependency and `[tool.bandit]` config from Task 2; the two-job structure from Task 1.
- Produces: a third top-level `jobs.security` job. Final workflow has exactly three parallel jobs: `test`, `typecheck`, `security`.

- [ ] **Step 1: Append the `security` job**

Add this job to `.github/workflows/ci.yml`, indented as a sibling of `test` and `typecheck` (i.e. under `jobs:`, at the same level), after the `typecheck` job:

```yaml
  security:
    runs-on: ubuntu-latest
    steps:
      - name: Check out the repository
        uses: actions/checkout@v5

      - name: Install uv
        uses: astral-sh/setup-uv@v8.2.0
        with:
          enable-cache: true

      - name: Install dependencies
        run: uv sync --frozen

      - name: Dependency vulnerability audit (pip-audit)
        run: |
          uv export --frozen --no-dev --no-emit-project --no-hashes -o requirements-prod.txt
          uvx pip-audit --no-deps -r requirements-prod.txt

      - name: Static security analysis (bandit)
        run: uv run --frozen bandit -c pyproject.toml -r src

      - name: Secret scan (gitleaks)
        run: |
          docker run --rm -v "${{ github.workspace }}:/repo" \
            ghcr.io/gitleaks/gitleaks:v8.30.1 dir /repo --redact --no-banner
```

Notes for the implementer:
- `requirements-prod.txt` is written into the runner workspace and never committed; it exists only for the lifetime of the job.
- `gitleaks dir` scans working-tree files (no git history, so the default shallow checkout is fine). It exits non-zero when a secret is found, which fails the step. `--redact` prevents printing any matched secret value; `--no-banner` trims noise.
- Ubuntu GitHub-hosted runners include Docker, so no Docker setup step is needed.

- [ ] **Step 2: Prove the final YAML is valid and has three parallel jobs**

Run:
```bash
uv run python -c "import yaml; d = yaml.safe_load(open('.github/workflows/ci.yml')); jobs = d['jobs']; assert set(jobs) == {'test','typecheck','security'}, jobs; assert all('needs' not in j for j in jobs.values()), 'jobs must be parallel'; print('OK three parallel jobs:', list(jobs))"
```

Expected: `OK three parallel jobs: ['test', 'typecheck', 'security']` and exit code 0.

- [ ] **Step 3: Re-run the security commands locally (bandit + pip-audit)**

Run:
```bash
uv run --frozen bandit -c pyproject.toml -r src
uv export --frozen --no-dev --no-emit-project --no-hashes -o /tmp/requirements-prod.txt && uvx pip-audit --no-deps -r /tmp/requirements-prod.txt
```

Expected: bandit reports no issues (exit 0); pip-audit reports `No known vulnerabilities found` (exit 0).

- [ ] **Step 4: Verify the gitleaks invocation**

The gitleaks scan runs from a Docker image and is verified in CI (Docker/gitleaks are not assumed present locally). If Docker **is** available locally, you may verify it directly:

Run (only if Docker is installed): `docker run --rm -v "$PWD:/repo" ghcr.io/gitleaks/gitleaks:v8.30.1 dir /repo --redact --no-banner`

Expected: `no leaks found` and exit code 0. (Verified during planning that `.env` and `.envrc` are gitignored and the only tracked env file is `.env.example`, which holds placeholders — so a clean pass is expected. If Docker is unavailable, skip to Step 5; CI will run this scan.)

- [ ] **Step 5: Commit**

```bash
git add .github/workflows/ci.yml
git commit -m "ci: add parallel security job (pip-audit, bandit, gitleaks)"
```

- [ ] **Step 6: Push and confirm all three jobs run green in CI**

```bash
git push -u origin HEAD
```

Then open the PR's Checks tab (or `gh run watch`) and confirm three separate jobs — `test`, `typecheck`, `security` — appear and pass. This is the only place the gitleaks scan and the parallel-execution behavior are exercised end-to-end.
